# Rim Combat

> 一个面向《上古卷轴 5》SE/AE 的近战战斗重构模组原型。  
> 它的目标不是简单提高数值或替换动作，而是把耐力、力竭、架势、破韧、处决和战技组织成一套更有节奏感的近战循环。

Rim Combat 围绕 MCO/BFCO、OAR、TrueHUD、PrismaUI 等生态进行设计，整体体验明显借鉴了《艾尔登法环》《只狼》《仁王》等作品中更强调资源管理与站位判断的近战思路，但仍然服务于 Skyrim 本身的战斗节奏与可扩展性。

---

## 总描述

Rim Combat 的核心目标，是让 Skyrim 的近战战斗从“持续输出”转向“资源、节奏与破绽管理”。

在这套设计里：

1. 攻击会真实消耗耐力，盲目连打会带来明显代价。
2. 耐力耗尽后会进入力竭，进攻和防守能力都会受到影响。
3. 近战交互会不断累积架势伤害，最终形成破韧窗口。
4. 破韧后可以继续压制，也可以寻找合适站位触发处决。
5. 战技则作为更高风险、更高收益的特殊攻击手段，服务于整套战斗循环。

简单来说，这个模组想做的不是“更花哨的动作”，而是“更有判断和博弈感的近战”。

## 模块描述

### 战技系统

战技系统是 Rim Combat 最核心的内容之一。玩家可以开启战技状态，并通过轻攻击 / 重攻击派生出不同的战技动作。部分战技还带有准备姿态，需要先进入准备阶段，再衔接后续动作，形成更接近动作游戏的输入体验。

目前战技菜单与 HUD 已经接通，玩家可以查看当前可用战技，并将战技分配给武器使用。

### 攻击耐力系统

Rim Combat 重做了近战攻击的耐力消耗逻辑。不同武器类型、不同攻击类型会带来不同的耐力负担，0 耐力时的攻击能力也会受到限制。

这一模块的目标是让每一次进攻都更有成本，而不是依赖无限制的连续压制。

### 架势条系统

模组新增了架势条这一资源。角色在受到攻击时会不断损失架势，架势归零后会进入破韧状态。

不同武器类型、攻击方式、格挡与限时格挡行为，都会影响最终造成的架势伤害。角色的最大架势值也会随生命值上限动态变化。

### 处决系统

当目标进入破韧状态后，处决系统会根据站位、距离、受害者种族和处决者武器类型，判断是否可以触发前处决或背刺处决。

当前这部分已经能完成基础的触发与判定，但整体仍然偏原型阶段，更多是在打磨机制本身，而不是最终演出。

### 力竭系统

当耐力被消耗到过低时，角色会进入力竭。力竭状态下，角色的攻防表现会受到明显影响，并且可以根据配置限制攻击行为。

这套系统的意义在于让战斗节奏出现真正的“失衡时刻”，而不是让双方一直维持相同强度的输出交换。

### 格挡 / 限时格挡系统

模组当前的格挡逻辑已经不只是单纯判断“是否成功格挡”，而是会根据武器类型映射不同的格挡强度，并进一步影响伤害减免、耐力消耗、架势减免与返架势效果。

限时格挡则是在普通格挡之上追加一个更高强度的防御窗口。成功触发时，可以参与后续的架势计算和事件分发。

当前这一模块的核心机制已经基本跑通，也已经可以稳定参与整套战斗循环测试。现阶段的重点主要转向数值调校、视觉反馈补完，以及等待后续模块接入时再扩展对外接口。

## 其他用户说明

### 当前已知限制

1. 同一个 FormID 的武器当前共享同一战技分配，不是按单个武器实例分别绑定。
2. 战技配置虽然已经支持从 JSON 读取，但目前还没有完全达到正式数据驱动状态。
3. NPC 战技相关内容目前还没有完成，因此当前体验主要以玩家侧为主。
4. 处决系统已经可以触发基础判定，但离最终演出效果和更多种族兼容还有一段距离。
5. 格挡系统的核心机制已经稳定，但反馈表现和与其他模块的联动接口还会继续扩展。

### 使用前说明

Rim Combat 当前依赖以下生态组件来获得完整体验：

| 组件 | 作用 |
| --- | --- |
| SKSE | 插件运行基础 |
| MCO / BFCO | 攻击动作框架 |
| OAR | 动作切换与条件分发 |
| TrueHUD | 架势条显示 |
| PrismaUI | 战技菜单与 HUD |

如果缺失其中的显示层依赖，部分核心逻辑仍然可能运行，但 HUD、菜单或视觉体验会明显不完整。

### 当前更适合怎样体验

如果你现在想试用 Rim Combat，更适合关注以下内容：

1. 不同武器类型下的攻击耐力消耗差异。
2. 0 耐力与力竭状态下的战斗压迫感是否成立。
3. 架势条削减、恢复、破韧与退出破韧的节奏是否自然。
4. 不同武器类型下格挡 / 限时格挡的减伤、耐力交换与架势反馈是否合理。
5. 面对面处决与背刺处决的触发稳定性。
6. 战技菜单、HUD 和战技切换体验是否顺畅。

### 当前不建议的使用方式

1. 不建议把当前版本视为正式发布版使用。
2. 不建议在长期正式存档中深度依赖当前版本。
3. 不建议现在就基于它做最终平衡判断，因为部分系统仍在快速迭代。
