# Rim Combat

> 一个面向《上古卷轴 5》SE/AE 的近战战斗重构模组原型。  
> 它的目标不是单纯抬高伤害或替换动作，而是把耐力、力竭、架势、破韧、处决和战技组织成一套更强调节奏、风险与站位判断的近战循环。

Rim Combat 围绕 MCO / BFCO、OAR、TrueHUD、PrismaUI 等生态进行设计，整体思路明显受到《艾尔登法环》《只狼》《仁王》等作品启发，但最终仍然服务于 Skyrim 自身的战斗节奏、模组生态与可扩展性。

---

## 总描述

Rim Combat 想做的，不是“让战斗更花哨”，而是“让每一次进攻、防守和失误都更有代价”。

在这套设计里：

1. 攻击会真实消耗耐力，连续压制不再几乎没有成本。
2. 耐力过低会进入力竭，角色的攻防表现都会明显下滑。
3. 近战命中会累积架势伤害，持续制造破韧窗口。
4. 格挡与限时格挡不只是防住攻击，还会参与耐力交换、减伤和架势博弈。
5. 目标破韧后可以继续压制，也可以寻找站位触发基础处决。
6. 战技作为更高风险、更高收益的动作分支，最终会承担整套循环里的爆发与变化。

当前版本已经可以跑通一条完整的近战原型循环：攻击消耗耐力，力竭影响攻防，受击累积架势，破韧进入可处决状态，格挡 / 限时格挡介入耐力与架势交换，战技菜单、HUD、背包联动分配面板、状态切换，以及由动画事件直接驱动的部分战技专属消耗 / 伤害处理，也已经接进实际战斗验证。受击侧也已经打通了一条基础链路：常规硬直继续交给 Maxsu Poise 与 Modern Stagger Lock，Rim Combat 则在原版 TryStagger 入口处覆写最终传入值，并在 ActionStaggerStart 阶段补上额外硬直等级与免疫判断。攻击侧的基础框架目前已经相对成型，下一阶段会更多转向“受击表现”和“特殊硬直”这条线。

不过，它仍然是一个明显处于持续迭代中的原型版本，不适合作为“正式完成版”来理解。

## 模块描述

### 攻击耐力系统

Rim Combat 已经重做了近战攻击的耐力消耗逻辑。当前耐力消耗会综合武器类型、攻击类型和武器重量进行计算，不同武器之间的进攻成本已经不再完全一样。

这一模块已经能稳定参与实际战斗循环，也是当前最接近“可长期调数值”的部分之一。

### 力竭系统

当耐力被消耗到过低时，角色会进入力竭。力竭状态下，攻击伤害、架势伤害、受击表现以及部分攻击权限都会受到影响。

这套系统已经能稳定参与节奏控制，目前的主要工作重心更多放在平衡和体验打磨，而不是补基础逻辑。

### 架势条系统

模组新增了架势条这一资源。角色在受到攻击时会持续损失架势，架势值归零后会进入破韧状态，并进一步接通处决逻辑。

当前架势系统已经完成了最大值计算、恢复逻辑、护甲对架势伤害的影响，以及与格挡、力竭、处决之间的基础联动。

### 格挡 / 限时格挡系统

当前格挡不再只是“挡住或没挡住”的二元判断，而是会依据武器类型映射格挡强度，进一步影响减伤、耐力消耗、架势伤害减免，以及对攻击者的返架势效果。

限时格挡则在普通格挡基础上追加了更高强度的窗口与持续时间效果，目前已经能稳定参与整套循环测试，也是当前完成度最高的核心模块之一。

现阶段这部分更需要的是数值继续微调、音效特效与击中反馈补完，而不是重做底层逻辑。

### 处决系统

当目标进入破韧状态后，处决系统会根据站位、距离、目标种类和武器类型，判断是否可以触发正面处决或背刺处决。

当前这部分已经能完成基础状态切换、角度判定、触发检查和原型级联动，但仍然明显偏向机制验证阶段。它现在更像是“处决框架已通”，而不是“最终演出已经完成”。

### 战技系统

战技系统目前已经具备基础菜单、HUD、武器分配、热键切换，以及准备态 / 启用态 / 执行中状态这套基础状态流程。当前分配界面采用背包联动的右侧面板形式，会随着左侧背包当前选中的物品自动刷新对应的武器信息与可分配战技。

在适配层，战技不再只依赖粗略武器分类，而是按武器重量、攻击方式、武器家族以及 Unique 指定武器这几类条件组合匹配。默认未填写的条件会按通配处理，而 Unique 则用于严格绑定特定武器，因此当前的分配结果已经比早期原型更接近真正可用的适配逻辑。

在触发层，战技的耐力消耗、部分伤害结算等数值开始由动画事件 payload 直接驱动，已经接入耐力与处决等处理。这意味着后续如果要扩展更多战技效果，重点会转向增加新的事件类型与接收端，而不是继续把大量固定字段塞进战技类本身。

同时，原本独立拆分的“伤害类型”思路已经被放弃。对当前这套系统来说，图事件和 payload 足以承担大部分战技 / 攻击扩展的表达需求，后续包括特殊硬直、霸体、受击降级等内容，也会优先沿用这套做法推进。

当前这一模块已经从单纯界面原型推进到“可实际验证的一整套基础框架”，但战技分配持久化、更多 payload 类型、NPC 侧逻辑、长列表筛选与更大规模列表下的菜单优化仍然没有真正收尾。

### 受击 / 特殊硬直扩展

Rim Combat 不会自己重做一整套常规轻硬直 / 大硬直 / 韧性基础系统，而是更倾向于借助 Maxsu Poise 与 Modern Stagger Lock 这类现有生态来承担“普通受击”部分。

目前已经接通的部分，是围绕原版 TryStagger 的一层重解释和覆写逻辑。实测可以确认：原版 TryStagger 的第二个参数并不是“最终硬直等级”，而是一次命中对应的硬直生命伤害百分比，原版内部会用这个值回写 staggerMagnitude。Rim Combat 现在会先在动画事件里缓存 targetset / targetmodify / immune 这类规则，再在 TryStagger detour 中根据这些规则修正传给原版的最终数值，最后在 ActionStaggerStart 阶段结合 staggerMagnitude 与额外的 RimCombat_StaggerLevel 图变量来识别 Strikefly、Knockaway、Knockdown 这类扩展硬直。

模组内部接下来真正要补的，是更偏动作设计层的受击扩展和表现收尾：例如更多战技或特定攻击过程中，对某一级别硬直的免疫、把某种受击下降一个级别，以及原版并不具备的挑飞、击飞、砸趴、仰倒这类特殊硬直的动作、反馈与边界处理。

这一部分已经不再只是设计和接口预留，而是完成了基础接线。后续仍会优先复用现有的图事件 / payload 体系，而不是再单独分出一套伤害类型或命中类型模块。

## 本轮检查

本轮针对当前工作区改动做了一次完整的代码检查，重点覆盖了 Hooks、事件分发、架势、处决、耐力、战技和特殊硬直这一条主链路。

目前已经确认的状态如下：

1. 图事件已经拆分为 RimStamina、RimPosture、RimWeaponArt、RimExecution、RimStagger 五条独立入口，事件分发与对应模块的 PayloadParse 接口已经对齐。
2. TryStagger 入口的覆写链路已经验证通过，原版第二参数按“硬直生命伤害百分比”处理，Rim Combat 当前会在 detour 中修正最终传给原版的值。
3. 当前工程可以正常通过构建检查，编辑器诊断也没有发现新的编译错误。

这轮检查里暂时没有发现新的 blocker，但有两条需要继续关注的已知风险：

1. Posture 的 Breakable 现在依赖图变量和动画 payload 显式控制。当前 BDI 默认把 RimCombat_PostureBreakable 注册为 false，而破防判定又直接读取这个值；如果某条动画路径没有正确发送 breakable|true 或后续复位事件，目标可能会一直处于不可破防状态。
2. Posture 的 damageCache 目前写入和清理使用了 mtx_damageCache，但在命中处理里读取时没有走同一把锁。现阶段这条链路主要按主线程时序工作，短期内通常不会直接表现为错误，但锁策略仍然不统一，后续继续扩展受击时序或异步任务时需要优先收口。

整体上，这次改动已经把“受击 / 特殊硬直”和“payload 驱动效果”这条线真正接到了现有战斗循环里；接下来更值得投入时间的不是继续加新入口，而是补齐 breakable、damage cache、特殊硬直表现和边界收尾这些稳定性问题。

## 其他用户说明

### 当前已知限制

1. 当前大量数值仍在快速调整，现阶段更适合把 Rim Combat 当成测试版原型来体验。
2. 大多数数值配置目前仍以 Settings.json 为主，游戏内设置菜单现阶段主要提供读取与重置，不是完整的设置界面。
3. 同一个 FormID 的武器当前共享同一战技分配，不是按单个武器实例分别绑定。
4. 战技系统虽然已经接通右侧分配面板、HUD、状态切换、解锁与分配流程，但部分数据与事件接口仍处于测试态，尚未达到完整正式的数据驱动状态。
5. NPC 战技相关内容目前还没有完成，因此当前体验仍然主要以玩家侧为主。
6. 常规受击 / 韧性体验的目标方案更偏向与 Maxsu Poise、Modern Stagger Lock 这类模组配合；Rim Combat 当前已经接管 TryStagger 入口上的规则覆写，自身更专注于特殊硬直和例外规则处理。
7. 处决系统已经可以触发基础判定与原型联动，但离最终演出效果和更广泛种族兼容还有明显距离。
8. 格挡系统的核心机制已经进入稳定测试阶段，但视觉反馈和与更多模块的联动接口还会继续扩展。
9. 战技分配保存、更多 payload 类型、战技点数消耗和更大规模列表下的菜单优化目前仍未完全收尾，更适合拿来测试流程而不是做最终养成判断。

### 使用前说明

Rim Combat 当前依赖或建议联动以下生态组件来获得更完整体验：

| 组件 | 作用 |
| --- | --- |
| SKSE | 插件运行基础 |
| MCO / BFCO | 攻击动作框架 |
| OAR | 动作切换与条件分发 |
| Maxsu Poise | 常规受击 / 韧性基础，并提供 TryStagger 调用来源 |
| Modern Stagger Lock | 提供 staggerMagnitude 阈值与常规硬直稳定化 |
| TrueHUD | 架势条显示 |
| PrismaUI | 战技菜单与 HUD |

如果缺失其中的显示层依赖，部分核心逻辑仍然可能运行，但 HUD、菜单或整体反馈会明显不完整。如果缺失常规受击相关模组，未来面向完整体验设计的硬直表现也会不完整。

### 当前更适合怎样体验

如果你现在想试用 Rim Combat，比较值得重点关注的是：

1. 不同武器类型下的攻击耐力消耗与压制成本是否合理。
2. 0 耐力与力竭状态下的战斗节奏是否真的形成压迫感。
3. 架势削减、恢复、破韧与退出可处决状态的节奏是否自然。
4. 不同武器类型下格挡 / 限时格挡的减伤、耐力交换与返架势反馈是否合理。
5. 正面处决与背刺处决的触发稳定性，尤其是近距离站位判断是否自然。
6. 战技菜单右侧伴随面板、自动跟随当前物品、准备态 / 启用态切换，以及武器分配流程是否顺畅。
7. 不同武器标签下的战技适配结果是否符合直觉，尤其是轻 / 常规 / 重、斩 / 刺 / 钝以及具体武器家族之间的区分。
8. 战技动作的专属耐力消耗、处决类事件、未来的特殊硬直事件，以及普通攻击耐力消耗之间的边界是否符合预期。
9. 如果联动 Maxsu Poise / Modern Stagger Lock，常规受击、TryStagger 覆写后的硬直衔接与霸体窗口的体验是否自然。
10. 修改 Settings.json 后再读取 / 重置设置时，整体行为是否符合预期。

### 当前不建议的使用方式

1. 不建议把当前版本视为正式发布版使用。
2. 不建议在长期正式存档中深度依赖当前版本。
3. 不建议现在就基于它做最终平衡判断，因为多个系统仍在快速迭代。
